Top N Queries

 

The TopN Queries test case collects the top 50 queries that consume high CPU usage, performed a high number of physical reads, and took the longest durations.  This test case is only available for SQL Server 2005 instances.  The objective of this test case is to identify the top slow or long running queries over a period of time.  Queries can run slowly because of excessive waits for acquiring locks, system resources waits (see the results of the WaitStats test case) that are blocking the queries, lack of proper indexes, missing or out-of-date statistics, non-optimal query plans and the list goes on and on…

The list of queries extracted out of this test case doesn’t constitute the set of bad queries on its own, but provide the list of first few queries you may want to focus on fine tuning when you also encounter performance issues out of your workload around the time the SQLRAP is performed.

Please note that this report shows only the queries that completed execution, but not the ones in the process of execution.

 

Data Collection

The test case queries the following DMVs to mine the top 50 queries.

 

sys.dm_exec_query_stats:  Consists of performance statistics for the query plans.  If the plan is removed from the cache, the corresponding row is eliminated from this DMV.

http://technet.microsoft.com/en-us/library/ms189741(SQL.90).aspx

 

There are some limitations with finding the top queries from sys.dm_exec_query_stats DMV.  The DMV reports wrong CPU utilization for queries that are executed in parallel. 

In high memory pressure conditions within cache, some of the top queries may have already evicted.  This would prevent you from seeing those queries in the report.  Not all query plans are cached, for example CREATE INDEX, DBCC checks, BACKUP operations etc…but they can be significant consumers of CPU, disk I/O and cause even blocking.  See Additional Checks and Information in this section to find out about more comprehensive reviews of expensive queries.

 

There are no configuration options necessary to run this test case.  It will run against the SQL instances as configured in the Change Settings\RAP\Targets area of the RADT.

 

Results

The results screen consists of two panes.  The top pane provides the list of SQL Server machine and instance names; the bottom pane renders a listing of Top 50 expensive queries in terms of high CPU, disk I/O and overall duration for the server name and instance name combination selected in the top pane.

Server/Instance Pane

This pane simply provides a list of the Server/Instances that were evaluated.  Selecting one of the Instances here will populate the Top Query Data in the bottom pane with that of the Instance selected.

Details Pane

This lower pane provides three tabs.

§  High CPU
Top 50 queries that were most expensive by CPU.

§  High Duration
Top 50 queries that were most expensive by Duration.

§  High Physical Reads
Top 50 queries that resulted in the most physical reads.

Each of these tabs will contain the following information for its associated top 50 queries:

§  Query Text: The actual text of the SQL query in the batch.  Since this can be extended text, it may be easier to view outside of the UI.  You can right-click, copy the cell, and paste it into another media for more effective viewing and analysis.  If there is a red X in the column, the string is too long for the UI, so you will want to look at this manually.

§   Query Plan: The SQL query plan for the whole batch. Since this can be extended text, it may not all show in the UI.  You can right-click, copy the cell, and paste it into another media for more effective viewing and analysis. If there is a red X in the column, the string is too long for the UI, so you will want to look at this manually.

§  Execution Count: The number of times that the plan has been executed since it was last compiled.

§  Total CPU: Total amount of CPU time, in milliseconds, that was consumed by executions of this plan since it was compiled.  Please note that this value could be wrong for parallel queries.

§  Avg CPU Time: The average CPU time in milliseconds for each execution of the plan.  Please note that this value could be wrong for parallel queries.

§  Total Duration: Total elapsed time, in milliseconds, for all completed executions of this plan.

§  Avg Duration: The average elapsed time in milliseconds for one execution of the plan.

§  Total Physical reads: Total number of physical reads performed by all executions of this plan since it was compiled.

§  Avg Physical Reads: The average number of physical reads performed by one execution of the plan.

§  Database Name: The name of the database for which context was in effect when the Transact-SQL statement corresponding to this plan was compiled, or the database name in which the query is executed.
