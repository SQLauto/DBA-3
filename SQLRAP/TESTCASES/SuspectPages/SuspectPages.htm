Suspect Pages

 

SQL Server contains logic to evaluate a page when it is retrieved that determines if the page is in a damaged state. If a page is damaged during a write operation, check values throughout the data page will not be in agreement. SQL Server will then write a message to the error log stating that a torn or damaged page has been detected.

 

These errors not only indicate the existence of corrupt data, but generally indicate that there is a hardware or packet filter driver problem on the system. If the problem is allowed to continue unchecked, irrecoverable damage can occur to the data on the system.

 

Data Collection

This rule queries this table for the events 1 (824 errors other than a bad checksum or a torn page), 2 (Bad checksum) and 3 (Torn page).

 

This rule is available only for SQL Server 2005 instances.

 

There are no additional configuration options for this test case. It executes against the SQL instances as configured in the Change Settings\RAP tab.  Data will be returned for this test case only if there are rule violations that occur. 

 

Results

The results screen for Suspect Pages reports the server name, instance name, Database Name, File Id, Page Id, Event Type, Error count (Number of times the error has occurred)  and Last Update Date

\The Event Type describes the type of the error, one of: an error 83 (disk error) or 84, bad checksum or torn page.

 

Detailed results (if applicable) can also be found in the SQL Results tab of the Issues screen.
