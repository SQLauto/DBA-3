Security Updates - MBSA

Failure to keep core infrastructure servers updated with the latest security updates and service packs leaves the entire environment at extreme risk to service outages, data loss and exposure, and other malicious activities. SQL Servers function as the underlying infrastructure component for many other servers and applications. The loss of the SQL Server infrastructure can be a business critical event.

 

Data Collection

The Security Updates test utilizes the Microsoft Security Baseline Analyzer (MBSA) 2.1 to remotely query each SQL Server targeted to determine if they are missing any important or critical security updates. A limitation to this method is that the tools computer and each server must have the Windows Update Agent 3.0 installed. This version of the agent is not installed by default in 2000 or 2003 operating systems. The target servers also need the Automatic Updates service started, although the service does not have to actually be used to download updates.

 

Note:   This test will skip a server if its Automatic Updates service is stopped, disabled or missing.   MBSA may be unable to scan servers with the Windows Firewall enabled.  Windows Server 2008 based servers have the firewall enabled by default and will reject remote scans without special steps taken.  MBSA itself must be configured to use a static port and the Windows Firewall on each target server must have an exclusion in place for this port.  Access over ports 135, and 139 or 445 is also required.

Results

Servers

¾  Server:  Displays the name of the server that was analyzed.

¾  Critical: Total number of critical security updates missing.

¾  Important: Total number of important security updates missing.

¾  Other: Total number of other updates missing.

¾  All: Overall total number of missing updates.

¾  MBSA Scan Status: Displays the scan result directly from MBSA.

¾  Error: Displays data collection errors.

 

Missing Updates

¡  Update Service: Displays the update service type.

¡  Update ID: Security bulletin number.

¡  Description: Title of the update.

¡  Severity: Displays the severity for the update.

¡  KB Article Installed: Displays the knowledge base article number.

 

All Updates

The “All Updates” tab shows a list of all updates that apply to the DC, whether installed or not installed.

 

Additional Checks and Information

¾  The open MBSA option in the review section opens MBSA and gives detailed information on the tests performed for each domain controller. This is a starting point for troubleshooting MBSA scan errors.

¾  Certain prerequisites must be met in order for MBSA 2.1 to evaluate a server. These include the following:

¡  Access to the Internet to download the latest catalog files. This can be done either through direct Internet access from the tools computer or by separately downloading the file from another computer and manually copying it to the tools computer.

   The security update catalog file can be downloaded from the following URL:

   Direct Download of the wsusscn2.cab File
http://go.microsoft.com/fwlink/?LinkId=76054

   The Windows Update redistribution catalog file can be downloaded from the following URL:

   Direct Download of the wuredist.cab File
http://update.microsoft.com/redist/wuredist.cab

   The authorization catalog file may also be required. It can be downloaded from the following URL:

   Direct Download of the muauth.cab File
http://go.microsoft.com/fwlink/?LinkId=43266

   The files should be copied to the profile of the account used to run the tool.  The exact path will depend upon the operating system.

   Windows Server 2003

C:\Documents and Settings\<username>\Local Settings\Application Data\Microsoft\MBSA\2.1\Cache

   Windows Server 2008

C:\Users\<username>\AppData\Local\Microsoft\MBSA\2.1\Cache

¡  The Windows Update Agent 3.0 or greater must be installed on the tools computer and every Server to be evaluated. This is an updated version of wuauclt.exe and is separate from the Automatic Updates service.

   How to Install the Windows Update Agent on Client Computers
http://technet.microsoft.com/en-us/library/bb932139.aspx

¡  The Automatic Updates service must be started on the target server. Servers do not have to perform automatic updates, but the service must be running.

¡  Even with all prerequisites in place, not all patching applications and mechanisms will allow the MBSA to correctly detect the installation of patches. In environments where the prerequisites cannot be met, or for whatever reason servers cannot be evaluated using the MBSA, alternative methods must be utilized.

¾  MBSA might display the following error after scanning a computer that is firewall protected:

“Cannot contact Windows Update Agent on target computer, possibly due to firewall settings”.

The following procedure describes how in Windows server 2008 the built-in advanced firewall can be configured to allow the MBSA scan to succeed.

¾  Configure Windows Server built-in firewall to allow MBSA to query the server:

¡  Windows server 2008 firewall can be managed through GPO to open the necessary ports for MBSA. It is also possible to remotely create firewall rules using netsh.exe. Before you can create firewall rules remotely on the option “remote firewall management” must have been enabled on all 2008 servers with the Advanced Firewall enabled. This can be achieved locally on each server or through a GPO.

   Locally on each server using netsh.exe:

netsh advfirewall firewall set rule group="Windows Firewall Remote Management" new enable=yes (no to disable the rule)

   Remote through GPO:

Computer Configuration\Policies\Windows Settings\Security settings\Windows Firewall with advanced Security\Windows Firewall with advanced Security\Inbound Rules:

Add the Predefined Rule “Windows Firewall Remote Management”

¡  Remotely create a new firewall rule to open the static COM+ port used by Windows Update:

   netsh -r <MachineName> (assuming the logged in user has Enterprise Admin access) advfirewall firewall add rule name="MBSA Static Port" dir=in protocol=TCP localport=<StaticPortNumber> action=allow

The rule is automatically enabled.

   To delete the rule when no longer needed use: netsh -r <MachineName> advfirewall firewall delete rule name="MBSA Static Port"

¡  On each firewalled DC configure the static COM+ endpoint for the Windows Update Agent.

You can configure the port using the following procedures:

   Startup script:

reg add HKLM\Software\Classes\AppID\{B366DEBE-645B-43A5-B865-DDD82C345492} /v Endpoints /t REG_MULTI_SZ /d ncacn_ip_tcp,0,<StaticPortNumber> /f

   Remote script:

Reg add \\<DCName>\HKLM\Software\Classes\AppID\{B366DEBE-645B-43A5-B865-DDD82C345492} /v Endpoints /t REG_MULTI_SZ /d ncacn_ip_tcp,0,<StaticPortNumber> /f

   Group Policy:

Use the 2008 Group Policy Preferences to add the 2008 registry key. 

   Note: In Server 2008 by default only the Trusted Installer has full access to this key. Before you are able to add this value you need to modify the permissions on the key first through a GPO or locally on each 2008 server.

¡  All other ports needed by MBSA are opened by default on a server with the built-in firewall enabled. Verify that the following firewall rules are enabled for MBSA to scan a remote DC:

   File and Printer Sharing (NB-Name-In) UDP port 137

   File and Printer Sharing (NB-Datagram-In) UDP port 138

   File and Printer Sharing (NB-Session-In) TCP port 139

   File and Printer Sharing (SMB-In)TCP port 445

   Windows Management Instrumentation (DCOM-In) TCP port 135
