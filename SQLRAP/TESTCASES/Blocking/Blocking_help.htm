Data Collection

The test case is designed to collect the blocking information in regular intervals set by the RAPID user (the default interval value is 10 seconds).  The data is collected over a period of time pre-defined in the settings; the default value is one hour.

The RAPID client uses T-SQL Scripts to query the following SQL Server DMVs to detect if any blocking exists, and to produce the blocking report for each execution of the test case.

In the simplest form, the script populates all sessions that are currently blocked, and navigates through the blocking hierarchy to find the head blocker for each blocking chain.  The script also performs several joins with the above DMVs to get the rest of the blocking details that are necessary to troubleshoot the blocking scenarios.

This test case has two configuration options which can be found in the Change Settings\General\Waitstats portion of the RADT.  They are:

-         Interval (in seconds):  How frequently does the script run upon execution of the test case.  Default is 60 seconds.

-         Duration (in minutes):  For what duration should the test case run upon execution.  Default is 60 minutes.

Results

The Blocking test case results screen has three panes.  The first pane shows the runtimes at which the test case found evidence of blocking for each server and instance combination.  The second pane provides the head blockers information for a selected server name/instance name as well as the runtime.  The third pane shows the rest of the details of the blocking chains: the sessions blocked, the sessions blocking, and the respective SQL statements the sessions are processing, and wait resources.  This pane also shows some of the notable events for instance actively running sessions in addition to waiting and provides the overall state of SQL Server at the time blocking was detected.

§  Head Blocker Pane

§  Head Blocker Session ID: If there are blocking sessions, head blocker is the session that is causing first blocking condition in a blocking chain.  There could be more than one head blocker session in a given data collection time, resulting in separate blocking chains.

§  Blocked Task count: Total number of blocked sessions recorded in the blocking chain started by head blocker session during the current iteration of the test case.

§  Total wait duration (milliseconds): Total wait time in milliseconds that is accumulated by all current blocked sessions in a blocking chain leading by the head blocker session since the blocking chain started.

§  Blocking Resource Wait type: The type of the wait resource, the possible wait types are “LOCK BLOCKING” if the blocking via locks, “PAGELATCH waits,” “PAGEIOLATCH waits,” or “Other”.

§  Average wait duration: The average wait time in milliseconds that a currently blocked session has waited in the given blocking chain originated by the head blocker session since the blocking chain started.

§  Max Wait Duration ms: The maximum wait time in milliseconds that any currently blocked session waited in the blocking chain started by the head blocker.

§  Max Blocking Chain depth: In each blocking chain started by the head blocker, the test case measures the number of sessions blocked in a cascading effect; for instance, if session 51 is blocking session 62 and session 62 is blocking session 70, the depth of this cascading blocking effect is 2.  This column displays the maximum depth out of all these scenarios in the current iteration for a given blocking chain.

§  Head Blocker stored procedure name: The name of the stored procedure that the head blocker session is currently executing in the case the session is executing a stored procedure.

§  Head Blocker proc ID: The object ID of the stored procedure that the head blocker session is currently executing.

§  SQL Statement: The SQL Statement that the head blocker session is currently executing or last executed.

 

§  Blocking chain details

§  Blocking Session ID: The session ID that is blocking the blocked session id

§  Wait resource: The resource for which the blocked session id is currently waiting.  The typical wait resource structure may have the information regarding the resource type, resource database id, resource associated entry within database for instance file id, object id etc….,

As an example the wait resource “RID:8:1:155:0” represent the blocked session is waiting to acquire a lock on the first row (Row id 0) on the page 155 on a database file 1 in the database 8.

§  Database Name: The database contect in use when the query is executed.

§  Object ID: The object id returned by the query text retrieved from sys.dm_exec_sql_text

§  Procname: The stored procedure name if the SQL command is currently waiting is a stored procedure.

§  Statement text: The SQL statement which that blocked session in the process of execution and is waiting on the grant of a lock on a resource in the wait resource column.
