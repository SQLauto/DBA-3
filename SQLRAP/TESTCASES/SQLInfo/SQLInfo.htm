SQL Info

This test case provides comprehensive information about the SQL Server instances that you are performing SQLRAP on.  We believe that the information SQLInfo test case gathers from SQL Server instances is very vital to determine the overall health and state of your SQL Server environment; this data can be used in conjunction with the results of any other test cases to evaluate the overall risk levels.

The other segment of this test case displays some critical configurations and state of all databases including system databases for a selected instance on the results TAB.

 

Data Collection

The test case uses T-SQL Query against the System DMVs, system tables and registry keys to extract the wide variety of data that SQLRAP presents on the results tab.  

Some of the DMVs and system functions that this test case uses include: sys.dm_os_performance_counters, sys.dm_os_memory_clerks and DATABASEPROPERTYEX

There are no configuration options necessary to run this test case.  It will run against the SQL instances as configured in the Change Settings\RAP\Targets area of the RADT.

 

Results

§  SQL Server Instance Information (Top pane of Results screen): This pane provides information pertinent to the whole SQL Server instance.  You will see one row per each SQL Server instance against which you executed this test case.

§  Server Name: Server Name is the Windows computer name on which the SQL Server instance is running.  For a clustered SQL Server instance, it is the virtual server name on which the SQL Server is running.

§  Instance Name: SQL Server instance name, for the default instance, the instance name is the server name.

§  Service Pack: SQL Server instance service pack information, the value ‘RTM’ represents the original release version, ‘SPn’ shows the service pack information.

§  Edition: SQL Server product edition, the edition determines the features and limitations of the respective SQL Server instance, for instance the maximum number of CPUs exposed to SQL Server.  Some of the possible edition values are “Standard Edition”, “Enterprise Edition” etc…

§  Build Number: SQL Server Build Number, the format is “Major.Minor.Build”.

§  NUMA Configuration: The number of nodes that the SQL Server instance is detected in NUMA (Non-Uniform Memory Access) configuration.  If the value is 1, either the NUMA architecture is not enabled or SQL Server instance didn’t recognize NUMA. 

For more details on “Understanding Non-Uniform Memory Access”, please see the following link: http://msdn.microsoft.com/en-us/library/ms178144.aspx

This information is not available in SQL Server 2000.

§  SQL Server Service Account: The Windows login account that has been used to run the SQL Server Database Engine Service.  Microsoft recommends using a domain user account with minimal rights for the SQL Server service, as the SQL Server service does not require administrator account privileges.

Please see the following links for more information on choosing properly secured windows service accounts for SQL Server related services:

Security Considerations for SQL Server installation: http://technet.microsoft.com/en-us/library/ms144228.aspx

Setting up Windows Service Accounts: http://msdn.microsoft.com/en-us/library/ms143504(SQL.90).aspx

§  SQL Server Agent Service Account: The Windows login account that has been used to run the SQL Server Agent service. 

SQL Server Agent service executes jobs, monitors and allows the automation of SQL Server administrative tasks.  Microsoft recommends not using the Network Service account to run either the SQL Server or the SQL Server Agent service. Local User or Domain User accounts are more appropriate for running these services.  The SQL Server Agent account must have administrator privileges if you create CmdExec and ActiveScript jobs that belong to someone other than a SQL Server administrator, or if you use the AutoRestart feature.

Microsoft recommends to restrict the execute permissions on cmdshell and ActiveScript jobs to only the members of sysadmin system role, but If the above features are requirements in your environment, consider using separate service accounts for the SQL Server and SQL Server Agent services.

§  Total Buffer Pool Size:  The total committed memory from the buffer pool (in kilobytes) at the time the SQLInfo test case is executed.  Note that this is not the total memory used by SQL Server but accounts for major SQL Server memory allocations including data, index, and plan cache pages etc... 

§  Target Server Memory: The total amount of memory (in kilobytes) that SQL Server can consume at the time the SQLInfo test case is executed.

§  Page Life Expectancy: Number of seconds a page stays in memory without being referenced at the time of execution of the SQLInfo test case. 

A buffer that has a 300-second page life expectancy will keep any given page in memory in the buffer pool for 5 minutes before the buffer pool flushes the page to disk unless a process references the page.  This counter can be helpful in determining whether you have a memory problem, giving you a reasonably accurate view of whether your server has memory pressure. Microsoft recommends 300 seconds as the minimum target for page life expectancy. If the buffer pool flushes your pages in less than 300 seconds, you probably have a memory problem.

§  Stolen Pages: Number of 8K size pages used for miscellaneous purposes (including procedure cache) collected at the time the SQLInfo test case is executed.

§  Data/Index Pages: Number of 8K size pages allocated as data and index pages at the time the SQLInfo test case is executed.

§  Lock Memory: Total amount of dynamic memory (in kilobytes) that SQL Server is using for locks at the time the SQLInfo test case is executed.

§  Workspace Memory: The total granted workspace memory (in kilobytes) for executing processes such as hash, sort, bulk-copy and index creation operations at the time the SQLInfo test case is executed.

§  Optimizer Memory: The total amount of dynamic memory (in kilobytes) that the SQL Server instance is using for Query Optimization at the time the SQLInfo test case is executed.

 

Database Information (Bottom pane of Results screen): pane you see database-specific data including state, configuration, and security information for a selected SQL Server instance on the SQL Server instance information pane.

§  Database Name: Name of the database, by default the rows are sorted by the database name attribute.

§  Database Status: State of the database at the time of execution of the SQLInfo test case.  The possible states include, "ONLINE","RESTORING","RECOVERING","RECOVERY PENDING", "SUSPECT", "EMERGENCY", and "OFFLINE".

For more information on the database states, follow the link: http://msdn.microsoft.com/en-us/library/ms190442(SQL.90).aspx

§  Recovery Model: The recovery model set to an individual database determines how much data can be recovered after a database crash event.  Each database can have a different recovery model based on how much data loss is tolerable during the restore operation.  The three recovery models are SIMPLE, BULK-LOGGED and FULL, out of which the FULL recovery model lets you to recover a database to a point-in-time.  The SIMPLE recovery model let you to recovery only to the last complete good database backup or differential backup.

For a complete understanding on recovery models, please see the following link: Overview of SQL Server recovery models: http://msdn.microsoft.com/en-us/library/ms189275(SQL.90).aspx

§  AutoCreateStats: Discloses if “Auto Create” stats database option is enabled or not.  When auto create statistics option is enabled on the database any missing statistics required by a query are automatically built during query optimization.  If the statistics are not used, the database engine automatically deletes those statistics.  Microsoft recommends enabling this option as a best practice.

§  AutoUpdateStats: When auto update stats option is enabled, any out-of-date statistics required by the query optimizer are automatically updated during the query optimization phase.   If this option is not set, the statistics must be updated manually.

§  AutoUpdateStatsAsync: When this option is set, queries that initiate an auto update stats operation will not wait for the statistics update operation to complete, but will instead use the old statistics.  Subsequent queries will use the updated statistics.  This option is not available for SQL Server 2000.  Microsoft recommends not enabling this option unless you notice the following characteristics.

If you experience several client queries time-out caused by the queries waiting on update statistics.

Predictable query response times are required, even at the expense of occasionally running queries with less efficient query plans because of outdated statistics.

 

For more information on Auto create\update statistics: http://msdn.microsoft.com/en-us/library/ms190397(SQL.90).aspx

§  PageVerifyOption: Determines the page verify option set to the database. Enabling this option discovers damaged pages caused by disk I/O path errors.  These errors can cause database corruption problems and are generally caused by power failures or disk hardware failures that occur at the time the page is being written to disk. Hardware induced corruption may not be detected until the next time DBCC CHECKDB is run. Data could also be lost if you have to restore to the last, clean backup.

Microsoft recommends to set the page verify option to CHECKSUM.  When CHECKSUM is enabled for the PAGE_VERIFY database option, the SQL Server Database Engine calculates a checksum over the contents of the whole page, and stores the value in the page header when a page is written to disk. When the page is read from disk, the checksum is recomputed and compared to the checksum value that is stored in the page header. This helps provide a high level of data-file integrity.  CHECKSUM is the default option for SQL Server 2005 and above, and not supported for SQL Server 2000 where the default and recommended option is “TORN_PAGE_DETECTION”.

§  AutoShrinkStatus: The AUTO_SHRINK option causes data and log files to be automatically shrunk if more than 25 percent of a file has unused space. This option can cause performance and concurrency issues during the shrink process. It can also cause index fragmentation after a shrink process has been performed.  Microsoft recommends disabling the Auto shrink option unless it’s absolutely required.

For best practice recommendations around Auto grow and Auto shrink options, please follow this link: http://support.microsoft.com/kb/315512

§  AutoClose: When enabled, this option causes the database to shut down cleanly and resources to be freed after the last database user exits. New connections to the database re-open it.  The AUTO_CLOSE option is not intended to be used for frequently accessed, highly concurrent shared databases. Thus, it should be enabled only for single-user desktop applications. Additionally, some of the performance issues experienced with AUTO_CLOSE prior to Microsoft® SQL Server 2005 have been resolved. However, performance issues can still occur in SQL Server 2005 because a database closure results in objects being cleared from the plan cache. 

§  Guest Users: Reveal if Guest user is enabled on the database.  The guest user allows a login without a corresponding user account in a database to access the database.  Microsoft recommends to drop (in SQL server 2000) or remove permission to access (in SQL Server 2005 and above) the user databases.  The system databases master and tempdb always have the guest user enabled, and this access cannot be dropped or revoked.

For more insight into guest user, please see the following link:
SQL Server 2005: http://msdn.microsoft.com/en-us/library/bb402861(SQL.90).aspx;
SQL Server 2000: http://msdn.microsoft.com/en-us/library/aa905195(SQL.80).aspx

§  DatabaseChaining: SQL Server databases can be configured to allow ownership chaining between specific databases or across all databases inside a single instance of SQL Server.  Cross-database ownership chaining is disabled by default for user databases.  Setting this option ON is not recommended unless all of the databases hosted by the instance of SQL Server must participate in cross-database ownership chaining and you are aware of the security implications of this setting.  Members of highly-privileged database roles in one database can use cross-database ownership chaining to access objects in databases external to their own.

§  CompatibilityLevel: Provides the compatibility level at which the database is set.  Setting the databases to a lower compatibility level should be done only as an interim aid during migration. Keeping the databases at a lower level indefinitely will disable some of the new features that later versions provide.

§  TrustworthyBit: This indicates whether the SQL Server instance trusts the database and contents within it.  By default the setting is OFF.  By disabling this option you can reduce certain threats exist as a result of attaching the database that contains malicious assemblies with external access or unsafe permissions and malicious modules that are defined to execute with high privileges.  This information is not available for SQL Server 2000 instances

http://msdn.microsoft.com/en-us/library/ms187861(SQL.90).aspx

§  ForcedParameterization: When the parameterization is set to FORCED, any literal value that appears in a DML statement is converted to a parameter during query compilation with few exceptions.  Though this is a very powerful feature that reduces the frequency of query compilations and recompilations, it must be used cautiously, and only when your workload testing determines that this option does not adversely affect your overall performance.  This information is not available for SQL Server 2000 instances.

For specific guidelines for using Forced Parameterization, please use this following link: http://msdn.microsoft.com/en-us/library/ms175037(SQL.90).aspx

§  is_read_committed_snapshot_On: The READ_COMMITTED_SNAPSHOT isolation level provides statement-level read consistency.  If this option is enabled at the database level, the transactions that are running under read committed isolation level use row versioning and do not acquire READ locks, which enhances concurrency.  Enabling row versioning must be evaluated carefully, as the performance impact may negate the actual advantages of this feature. 

§  Snapshotisolationlevel: The SNAPSHOT isolation level provides the transactional level read consistency.  Every statement within a transaction running under snapshot isolation level sees only committed changes that occurred before the start of the transaction. 

In systems that have a high percentage of data changing, enabling the use of the snapshot isolation level may have a negative effect on overall performance, because the overhead of creating and managing the previous versions of a row can slow down transactions, particularly when tempdb or the disk subsystem is already close to being a system bottleneck. In this situation, the performance cost of enabling this infrastructure may not be worth the value of reporting against the real-time data, especially because any reporting will likely add even more load to an already busy system.

For a comprehensive review on these SQL Server 2005 snapshot isolation level features, please read the below whitepaper. http://msdn.microsoft.com/en-us/library/ms345124.aspx (The snapshot isolation is not available for SQL Server 2000 instances.)

§  Publisher: Will let you know if the database is a publication database in a transactional or snapshot replication topology.

§  Subscriber: Will let you know if the database is a subscription database in a transactional or snapshot replication topology.

§  Distributer:  Will let you know if the database is a distribution database in a transactional or snapshot replication topology.

§  ServiceBrokerEnabled: Shows if service broker message delivery is activated in the database.  By default restored or attached databases have the broker disabled.  This information is not available in SQL Server 2000.

§  Log_reuse_wait: Provides the potential reason for a transaction log currently awaiting reuse.  This information is very useful in troubleshooting transaction log truncation problems that may lead to unexpected transaction log file growth.  This information is not available in SQL Server 2000.

Factors that can delay transaction log truncation: http://msdn.microsoft.com/en-us/library/ms345414(SQL.90).aspx
