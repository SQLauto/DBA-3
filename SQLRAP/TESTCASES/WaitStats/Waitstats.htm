Waitstats

The SQL Server wait stats play a significant role in performance and tuning methodology by helping to identify the bottleneck areas and best opportunities to improve performance.  An analysis on wait stats indicates where SQL Server is spending its time waiting.  The wait stats data often can be used in conjunction with the performance data collected using other tools like performance monitor and SQL Server Profiler to quickly nail down specific problem areas without guess work.  This method significantly reduces your time in troubleshooting the problematic areas and allows more time to be focused on fine tuning efforts.  For instance, high wait times on page I/O latches could correlate with high disk response time performance counter values, and the DBA can quickly turn attention to the SQL Server Profiler output to find the top queries that are causing a high number of reads, while engaging the disk administrators to investigate the disk I/O system issues at the same time.

 

Another typical example is when a clustered index is created on an identity column.  If the table is a target for multiple inserts from several concurrent connections, the application may experience a slower INSERT operation rate.  If you collect the wait stats information during that time, you may notice waits on the wait types “PAGELATCH_EX” and PAGELATCH_SH”.  This happens because several concurrent INSERT rows fall into the same page or adjacent pages as a result of the choice of the cluster index key.  In this scenario the wait stats chart produced by the “waitstats” test case is handy to identify these latch waits behaviors over a period of time and patterns as well helping you identify the bottleneck areas promptly.

 

Data Collection

The wait stats data is collected using a T-SQL script, that queries the system DMV “sys.dm_os_wait_stats” in SQL Server 2005 or the “DBCC SQLPERF(‘Waitstats’)” command in SQL Server 2000, to get the cumulative wait times since the SQL Server started or wait stats were cleared manually using DBCC SQLPERF(‘sys.dm_os_wait_stats’, CLEAR).  The script computes the delta wait time (the time difference in milliseconds between the current run values and previous run value) on every wait resource and outputs those values to the RAPID client for each interval of execution.  Before exiting an iteration, the values for the current wait stats are stored in a table created in tempdb so that the values can be used as previous run values for the next iteration.  The table created in tempdb is dropped at the end of the final iteration.  The table created in tempdb always contains the wait stats values that are collected for the previous execution only; that way the table size always fixed irrespective of how long you run this test case.

The test case excludes some of these wait types ‘LAZYWRITER_SLEEP','SQLTRACE_BUFFER_FLUSH','BROKER_TASK_STOP','WAITFOR','RESOURCE_QUEUE','Total','SLEEP' represent idle cycles of the system SPIDs like lazy writer etc… 

 

This test case has two configuration options which can be found in the Change Settings\General\Waitstats portion of the RADT. 

-         Interval (in seconds):  How frequently does the script run upon execution of the test case.  Default is 60 seconds.

-         Duration (in minutes):  For what duration should the test case run upon execution.  Default is 60 minutes.

 

Results

The test case computes the delta (change in) wait time between two successive iterations and plots a graph with the data collection time on the X-Axis against the delta wait time on the Y-Axis for the top 5 wait types.  The test case also presents individual charts for the top 5 wait types.  The top 5 wait types are chosen based on the average delta wait time that SQL Server experienced on each wait type during the data collection period.

§  Charts Tab
The Charts tab plots a consolidated graph with top 5 wait types (based on top Average delta wait times recorded during the test case data collection) with the interval times on X-AXIS and delta wait time in milliseconds on Y-AXIS.  There will be one consolidated chart per each SQL Server instance against which you executed the test case.

§  Instance Tab
The Instances tab provides individual charts for a selected wait type and SQL Server instance on the right pane.  It also displays the actual runtime stamps, and delta wait time values for the above selected combination.  Simply select the Instance and Waitstat you wish to view from the selection menu at the top.
