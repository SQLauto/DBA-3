DBCC Verification - Recent

The DBCC Verification test case reports all user databases which have not had a DBCC CHECKDB run against them in the last seven days.  This test case only returns results when run against instances running SQL Server 2005 or greater.

 

DBCC CHECKDB should be run regularly. The frequency in which DBCC CHECKDB should be run against any particular database depends largely on the individual business needs and the importance of the information in the database. However, at a minimum, DBCC CHECKDB should be run against all production databases at least once a week.  In addition, the results should be reviewed as soon as possible after execution completes in order to identify and resolve any errors before they become critical.

 

DBCC CHECKDB should also be run on a regular basis to proactively identify inconsistencies in data structures and other types of corruption.

Failure to resolve the problems detected by DBCC CHECKDB in a timely manner can lead to significant data loss and to the server failing to respond.

 

If DBCC CHECKDB uncovers any errors, it will specify the minimum repair level that should be used to fix the problem. Note that running DBCC CHECKDB with the repair option might not resolve the issue, depending on the type of damage uncovered. This could potentially result in a significant loss of data. 

 

DBCC CHECKDB in Microsoft® SQL 2005 uses an internal database snapshot to perform the checks in order to avoid blocking and concurrency problems. However, because SQL Server 2005 performs much more extensive checks than earlier versions, it may take considerably longer to complete. Thus, Microsoft recommends that the PHYSICAL_ONLY option be used for frequent checks on production databases.

 

Data Collection

The DBCC Verification test case data is collected using a T-SQL script which queries the system tables as follows:

·         DBCC Verification (sql_dbcc_dbinfo_checkdb_run_verification.sql):

§  For SQL Server 2005 and greater, builds a temp table to house the results of DBCC DBINFO WITH TABLERESULTS calls.  Runs this command against all user and system databases except tempdb, and extracts the dbi_dbccLastKnownGood (the date of the last clean DBCC CHECKDB) value into another temporary table.  Reports any databases which haven’t had a clean DBCC CHECKDB in at least seven days as reflected by the system clock.  Will also note if a database has never had a clean DBCC run against it.

There are no additional configuration options for this test case. It executes against the SQL instances as configured in the Change Settings\RAP tab.  Data will be returned for this test case only if there are rule violations that occur. 

 

Results

The results screen for DBCC Verification reports the server name, instance name, database id, database name, and days since last DBCC CHECKDB for each database on the instance for which more than seven days have elapsed since the last clean DBCC CHECKDB.

 

Detailed results (if applicable) can also be found in the SQL Results tab of the Issues screen.
